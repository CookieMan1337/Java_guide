# Глава 1. Введение в Spring Framework и Spring Boot

## 1. С чего начинается проблема

Когда человек знает только Java Core, у него обычно складывается вполне логичное впечатление: чтобы написать программу, достаточно создать классы, вызвать методы, собрать всё через `new` и запустить `main`. Для небольших учебных примеров этого действительно хватает. Но как только приложение начинает походить на реальное, возникает совсем другой набор задач: нужно поднимать HTTP-сервер, читать настройки, подключаться к базе данных, работать с транзакциями, логировать события, обрабатывать ошибки, разделять код на слои и не запутаться в зависимостях между десятками классов. Именно в этой точке “чистая Java” остаётся основой, но уже перестаёт быть достаточным уровнем абстракции для удобной прикладной разработки.

Исторически Spring появился как ответ на сложность ранней enterprise-разработки на Java. В официальном обзоре Spring Framework прямо сказано, что проект возник в 2003 году как реакция на сложность ранних спецификаций J2EE. Это важный момент для понимания: Spring был придуман не ради моды и не ради “магии аннотаций”, а как попытка сделать разработку бизнес-приложений проще, гибче и понятнее.

Главная практическая проблема, которую решает Spring, — не “как написать ещё один класс”, а “как собрать приложение из множества компонентов так, чтобы оно было поддерживаемым”. Если каждый объект в системе сам создаёт себе все зависимости, код быстро становится жёстко связанным. Такой проект трудно тестировать, трудно менять, трудно расширять и особенно трудно сопровождать, когда он вырастает из десяти классов в несколько сотен. Spring предлагает другой подход: инфраструктура берёт на себя управление созданием и связыванием объектов, а разработчик сосредотачивается на бизнес-логике. 

Поэтому знакомство со Spring лучше воспринимать не как изучение одной библиотеки, а как вход в определённую модель разработки. В этой модели приложение — это не хаотичный набор Java-классов, а управляемая среда, где у компонентов есть жизненный цикл, у зависимостей есть понятный способ внедрения, у конфигурации есть единая точка управления, а у инфраструктурных задач есть готовые механизмы. Именно эта идея будет проходить через весь дальнейший курс.

## 2. Что такое Spring Framework

Spring Framework — это фундаментальный каркас для разработки Java-приложений. В официальной документации подчёркивается, что Spring Framework разделён на модули, и приложение может использовать только те части, которые ему действительно нужны. Когда в разговоре говорят просто “Spring”, это иногда означает всю семью Spring-проектов, но в строгом смысле Spring Framework — это именно базовая основа, с которой всё началось.

В центре Spring Framework находится контейнер, обычно представленный через `ApplicationContext`. Документация формулирует это очень прямо: `ApplicationContext` представляет Spring IoC container и отвечает за создание, конфигурацию и сборку бинов. Если перевести это на простой язык, контейнер — это управляющий механизм, который знает, какие объекты есть в приложении, как их создать и кому что передать для работы.

Здесь появляется ключевое слово — IoC, Inversion of Control, то есть инверсия управления. Без Spring объект обычно сам создаёт то, что ему нужно: например, сервис внутри себя делает `new Repository()` и жёстко привязывается к конкретной реализации. В Spring подход другой: объект лишь объявляет свои зависимости, а контейнер подставляет их при создании. Официальное описание dependency injection как раз и говорит о том, что объект определяет свои зависимости через конструктор, фабричный метод или свойства, а контейнер внедряет их во время создания бина.

Из этого вытекает практическая ценность Spring Framework. Классы становятся слабее связанными, тестировать их проще, менять реализации безопаснее, а инфраструктурные задачи можно подключать централизованно. Поэтому Spring Framework — это в первую очередь не “набор аннотаций”, а инженерный способ организовать приложение так, чтобы оно не разваливалось по мере роста. Чем раньше это понять, тем легче будет воспринимать все следующие темы.

## 3. Что такое Spring Boot и почему он почти всегда идёт рядом со Spring

Если Spring Framework — это фундамент, то Spring Boot — это удобная надстройка над этим фундаментом. Официальная документация Spring Boot определяет его как средство для создания stand-alone, production-grade Spring-based applications. Там же сказано, что Boot использует opinionated view of the Spring platform, чтобы разработчик мог начать работу с minimum fuss, то есть с минимальной ручной вознёй вокруг конфигурации.

Очень важно не путать Framework и Boot. Spring Framework отвечает за базовые механизмы: контейнер, бины, внедрение зависимостей, инфраструктурные модули. Spring Boot не заменяет эти механизмы, а делает их удобнее в повседневной разработке. Иными словами, Boot не “другая технология”, а способ быстрее и проще пользоваться Spring в реальных проектах. 

Почему Boot стал настолько популярным, становится понятно почти сразу, если посмотреть на его цели. В официальном описании перечислены: быстрый старт, разумные значения по умолчанию, встроенные серверы, безопасность, метрики, health checks и внешняя конфигурация. То есть Boot решает не только задачу “как быстро поднять приложение”, но и задачу “как сделать так, чтобы это приложение было удобно запускать, сопровождать и эксплуатировать”.

Для новичка Spring Boot особенно ценен тем, что убирает с пути ранний технический шум. Вместо того чтобы сначала разбираться с большим количеством ручной конфигурации, можно быстро получить рабочий проект, запустить его, увидеть результат, а уже потом постепенно разбирать, какие механизмы стояли за этим удобством. Это хороший учебный путь: сначала увидеть цельную картину, а потом послойно разбирать её устройство.

## 4. Что именно входит в мир Spring

Важно понимать, что Spring — это не один модуль и не один jar-файл. В официальном обзоре Spring Framework сказано, что он разделён на модули, а в центре находятся модули core container с моделью конфигурации и механизмом dependency injection. Дальше поверх этого фундамента Spring Framework даёт поддержку для разных архитектурных стилей: messaging, transactional data, persistence и web.

Если говорить на уровне первых ориентиров, то у Spring Framework есть как минимум несколько больших смысловых зон. Первая — это ядро: контейнер, бины, конфигурация, профили, свойства окружения. Вторая — веб-слой, где живут контроллеры, маршрутизация, обработка HTTP-запросов и форматы данных. Третья — доступ к данным, транзакции и интеграция с persistence-решениями. Четвёртая — вспомогательные механизмы вроде AOP, тестирования и интеграции с сообщениями. Такая карта важна сейчас не ради глубины, а чтобы читатель видел, что Spring — это экосистема, а не один “веб-фреймворк”.

Для веб-разработки в рамках Spring Framework есть как минимум два заметных направления. Первый — Spring Web MVC, классический веб-фреймворк на основе Servlet API, который присутствует в Spring с самого начала. Второй — Spring WebFlux, реактивный веб-стек, появившийся позже для неблокирующих сценариев. Для первой вводной главы достаточно зафиксировать простую мысль: когда вы делаете обычное серверное веб-приложение, чаще всего на старте вы будете иметь дело именно со Spring MVC.

Отдельно нужно сказать про семью Spring-проектов в более широком смысле. Spring Framework — это фундамент, но поверх него существует много других проектов: Spring Boot, Spring Data, Spring Security, Spring Cloud и другие. Именно поэтому в вакансиях и на проектах часто говорят просто “нужен Spring-разработчик”, хотя по факту речь идёт о работе сразу с несколькими частями экосистемы. Для новичка полезно с самого начала понимать эту разницу: есть ядро, а есть инструменты и расширения вокруг него.

## 5. Какие задачи Spring решает в реальной разработке

Первая большая задача Spring — управление зависимостями между компонентами приложения. Как только у вас появляются контроллеры, сервисы, репозитории, клиенты внешних API, мапперы и конфигурационные классы, приложение превращается в граф взаимосвязанных объектов. Ручное создание и сборка этого графа быстро становятся неудобными. Контейнер Spring как раз и нужен для того, чтобы централизованно создавать, конфигурировать и связывать такие компоненты.

Вторая задача — стандартизация прикладной структуры. Spring не заставляет писать бизнес-логику каким-то единственным способом, но даёт устойчивые архитектурные привычки: отделять веб-слой от сервисного, сервисный — от доступа к данным, конфигурацию — от бизнес-кода, инфраструктуру — от предметной области. Это не магическое правило самого фреймворка, а естественное следствие того, как устроены контейнер, конфигурация и работа с бинами. Поэтому Spring-проекты разных команд часто выглядят похоже даже тогда, когда писались разными людьми.

Третья задача — работа с инфраструктурой приложения. Реальное серверное приложение почти никогда не ограничивается только обработкой методов. Оно должно слушать HTTP-порт, читать внешние настройки, логировать работу, подключать безопасность, мониторинг, health checks, а в дальнейшем — работать с базой, сообщениями и интеграциями. В описании Spring Boot отдельно подчёркивается, что он предоставляет common non-functional features, включая embedded servers, security, metrics, health checks и externalized configuration. Это и есть та часть разработки, которую новичок часто недооценивает, пока не увидит живой проект.

Четвёртая задача — ускорение старта разработки. Boot предлагает стартеры, то есть готовые наборы зависимостей под типовые сценарии. В официальной документации сказано, что starters — это удобные dependency descriptors, которые добавляют набор библиотек, обычно нужных для конкретного типа приложения. Например, для обычного веб-приложения достаточно подключить `spring-boot-starter-web`, и проект получает согласованный набор зависимостей для HTTP-слоя и встроенного веб-сервера.

## 6. Из чего обычно состоит Spring Boot-приложение

Если смотреть на минимальное приложение глазами новичка, внутри него есть несколько базовых частей. Обычно это точка входа, где запускается приложение, набор управляемых компонентов, например контроллеров и сервисов, конфигурация, которая может лежать в `application.yml` или `application.properties`, и зависимости в Gradle или Maven. Всё это вместе образует не просто набор файлов, а описание того, как приложение должно быть собрано и запущено контейнером Spring.

Точка входа в Spring Boot-приложение обычно выглядит очень компактно. Это возможно именно потому, что Boot берёт на себя значительную часть стартовой настройки и запускает контекст приложения за вас. Новичка здесь может удивить, насколько мало кода нужно для старта: по объёму это часто меньше, чем “ручной” запуск аналогичной инфраструктуры без фреймворка. 

Простейший стартовый класс может выглядеть так:

```java
import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;

@SpringBootApplication
public class DemoApplication {
    public static void main(String[] args) {
        SpringApplication.run(DemoApplication.class, args);
    }
}
```

Такой класс не показывает всей внутренней механики, но он показывает главное: у Spring Boot есть понятная точка входа, а запуск приложения сводится к поднятию Spring-контекста. Именно из-за этого Boot воспринимается как очень удобная стартовая оболочка над Spring Framework. 

Если добавить к такому проекту веб-зависимость, можно почти сразу получить HTTP-обработчик. Для первых шагов этого достаточно, чтобы связать абстрактную теорию с реальным результатом: приложение стартует, слушает порт и отвечает на запрос.

```java
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;

@RestController
public class HelloController {

    @GetMapping("/hello")
    public String hello() {
        return "Hello, Spring!";
    }
}
```

Смысл этого примера не в том, чтобы запомнить аннотации, а в том, чтобы увидеть: в Spring приложение описывается через компоненты, а инфраструктура вокруг них поднимается фреймворком. Позже вы подробно разберёте, что такое контроллер, бин, сканирование компонентов и автоконфигурация. Сейчас важно увидеть общую картину. 

## 7. Как начать работу без лишней путаницы

На сегодняшний день самый простой официальный путь старта — Spring Initializr. В документации Initializr он описан как расширяемый API для генерации JVM-проектов, а в quickstart на spring.io предлагается начать именно с `start.spring.io`, выбрать новый Spring Boot project и добавить нужные зависимости. Для новичка это лучший вариант, потому что он убирает рутинную ручную сборку каркаса проекта и даёт сразу рабочую основу. 

Для Spring Boot нужен Java SDK 17 или выше. Это прямо зафиксировано в официальных требованиях и в разделе установки Spring Boot. Кроме того, Boot нормально работает с классическими Java-инструментами и рекомендует использовать build tool с dependency management, например Maven или Gradle. Для вашего курса это хорошо совпадает со стеком: Java 17+ и Gradle — нормальная и современная точка входа. 

Если идти по самому короткому учебному маршруту, то старт выглядит так: создать проект через Spring Initializr, выбрать Java 17+, выбрать Gradle, добавить зависимость `Spring Web`, открыть проект в IDE, запустить главный класс приложения и проверить, что сервер поднялся. Затем можно добавить первый контроллер и отправить запрос на локальный адрес. Это и есть тот момент, когда абстрактная фраза “Spring Boot ускоряет старт” превращается в наблюдаемую реальность. 

Ещё одна сильная сторона Boot в том, что приложение можно запускать как обычный исполняемый jar через `java -jar`. Для разработчика это означает более простой жизненный цикл: собрать, запустить, проверить локально, положить в контейнер, деплоить дальше. На старте не нужно погружаться в тяжёлую инфраструктуру развертывания — достаточно понять, что Boot-приложение ощущается как обычное Java-приложение, но с очень мощной инфраструктурой внутри. 

## 8. Что новичку особенно важно не перепутать

Первая типичная ошибка — думать, что Spring и Spring Boot это одно и то же. Нет. Spring Framework — это база и основные механизмы. Spring Boot — это удобный способ быстро собрать и запустить приложение на этой базе. Если не различать эти уровни, дальше очень легко запутаться в темах про бины, контекст, авто-конфигурацию и стартеры. 

Вторая ошибка — считать, что Spring “сам всё делает” и поэтому понимать внутреннее устройство не нужно. На первом шаге действительно может показаться, что достаточно поставить несколько аннотаций, и приложение оживёт. Но это ощущение обманчиво. Spring Boot скрывает много рутинной настройки, однако не отменяет базовых принципов контейнера, конфигурации и dependency injection. Если их не понять, любое нестандартное поведение быстро превращается в непонятную “магию”. 

Третья ошибка — пытаться учить Spring по отдельным аннотациям. Такой путь выглядит легко только в первые дни. Человек запоминает `@RestController`, `@Service`, `@Configuration`, `@SpringBootApplication`, но не понимает, какое место каждая из них занимает в общей системе. Гораздо правильнее строить знание от общей модели: есть контейнер, есть управляемые объекты, есть зависимости, есть конфигурация, и есть Boot, который автоматизирует типовые решения. Тогда новые инструменты укладываются в понятную схему. 

И ещё один важный момент: Spring не заменяет знание Java. Он строится поверх него. Если человек плохо понимает классы, интерфейсы, исключения, коллекции, потоки, HTTP и основы работы приложения, то Spring не “спасёт”, а только добавит новый слой сложности. Но если база есть, то Spring становится очень мощным усилителем, потому что позволяет писать большие приложения намного системнее и быстрее. 


## 9. Spring Boot Starters

Spring Boot ввёл понятие **стартеров** (starters). Это специальные зависимости, которые включают в себя всё необходимое для определённой задачи.

Примеры:

* `spring-boot-starter-web` — веб-приложения (Tomcat, Spring MVC, Jackson).
* `spring-boot-starter-data-jpa` — работа с базами данных через Hibernate.
* `spring-boot-starter-security` — аутентификация и авторизация.

Без стартеров вам пришлось бы добавлять каждую библиотеку вручную. Со стартером достаточно подключить одну зависимость.

Gradle (Groovy):

```groovy
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-web'
}
```

Gradle (Kotlin):

```kotlin
dependencies {
    implementation("org.springframework.boot:spring-boot-starter-web")
}
```

## Инструменты для старта

Для работы со Spring Boot нужны:

1. **JDK 17+** — официально поддерживаемая версия (с долгосрочной поддержкой).
2. **IDE** — IntelliJ IDEA (рекомендуется), Eclipse или VS Code.
3. **Система сборки** — Gradle или Maven (Gradle чаще быстрее и гибче).
4. **Git** — для контроля версий.
5. **Docker** (опционально) — для запуска баз данных и сервисов.

## Стандартная структура проекта

Spring Boot придерживается стандартной структуры:

```
src/
 ├─ main/
 │   ├─ java/com/example/demo/   (код приложения)
 │   └─ resources/               (application.yml, статические файлы)
 └─ test/                        (тесты)
```

Главный класс приложения обычно лежит в корне пакета `com.example.demo` и содержит аннотацию `@SpringBootApplication`.

---

## Аннотация @SpringBootApplication

Эта аннотация объединяет сразу три других:

* `@Configuration` — помечает класс как конфигурацию Spring.
* `@EnableAutoConfiguration` — включает авто-конфигурацию.
* `@ComponentScan` — сканирует пакеты на наличие компонентов (`@Service`, `@Repository`, `@RestController`).

То есть одной аннотацией вы активируете всю инфраструктуру.

## Запуск приложения

Приложение можно запустить несколькими способами:

1. Через IDE — кнопкой **Run** в классе `DemoApplication`.
2. Через Gradle:

   ```bash
   ./gradlew bootRun
   ```
3. С помощью jar:

   ```bash
   ./gradlew build
   java -jar build/libs/demo-0.0.1-SNAPSHOT.jar
   ```

Разница между `bootRun` и `java -jar` в том, что `bootRun` удобен на этапе разработки (горячая перезагрузка классов), а jar используется для деплоя.

---

## Настройки и порты

Все настройки Spring Boot хранятся в `application.properties` или `application.yml`. Например, чтобы сменить порт:

```yaml
server:
  port: 8081
```

Это не требует изменения кода — только конфигурации.

---

## Встроенные серверы

Spring Boot поддерживает несколько встроенных серверов:

* **Tomcat** (по умолчанию).
* **Jetty**.
* **Undertow**.

Tomcat — универсален и подходит для большинства случаев. Jetty и Undertow используют, когда важна производительность или особенности протоколов.

## Итог

Spring Boot — это стандарт де-факто для разработки приложений на Java. Он убирает рутину, позволяет сосредоточиться на логике и помогает стартовать за минуты.

Мы познакомились с ключевыми понятиями: авто-конфигурация, стартеры, Initializr, структура проекта и запуск. Эти знания — фундамент для дальнейших тем.

Важно помнить: Spring Boot не магия. Всё, что он делает, можно было бы настроить вручную в чистом Spring. Но за счёт автоматизации и удобных соглашений разработка становится быстрее и приятнее.
